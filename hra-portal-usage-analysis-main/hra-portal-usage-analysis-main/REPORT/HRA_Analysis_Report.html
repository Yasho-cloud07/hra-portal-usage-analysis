Registration User Interface (RUI) — Trends & Behavior

What is the RUI?

The RUI allows biomedical researchers to spatially register tissue samples onto 3D reference organ models. When a researcher moves or places a tissue block in the 3D viewer, the browser fires collision detection API calls in real-time. The workflow: select organ → load extraction sites → iteratively position tissue block → confirm spatial placement. CloudFront logs this entire session through four key endpoint families.

Fig 1 — RUI Monthly Usage: Event Volume & User Growth
🔍 Key Insight — Event Volume The November 2024 spike (189K events) is the most significant signal in the RUI dataset — nearly 10× the average monthly volume. This strongly correlates with a HuBMAP Consortium event, workshop, or a coordinated data release that drove concentrated tissue registration activity. The secondary spike in September 2025 (~52K events) shows the platform is actively growing. Unique users have grown from ~300/month (late 2023) to a steady 1,000–1,400/month by late 2025.
Fig 2 — RUI Feature Categories: What Are Researchers Actually Doing?
🔍 Key Insight — Tissue Registration Funnel Tissue Collision Check (436K events, 53%) dominates because every mouse drag in the 3D placement tool fires a real-time collision API call — it is a proxy for active engagement, not passive viewing. The workflow funnel is: Collision (436K) → Extraction Site (224K) → Spatial Placement (93K) → Tissue Block Query (65K). The 4.7:1 collision-to-placement ratio is expected and healthy — it reflects the iterative, exploratory nature of spatial tissue registration. Users try multiple positions before committing.

Fig 3 — RUI Organ Registrations: Which Organs Are Researched Most?
🔍 Key Insight — Organ Distribution The Large Intestine (female) leads organ registrations with ~4,081 landmark calls, followed by Spleen (female), Kidney (male), and Heart (male). This pattern reflects active HuBMAP research priorities in gastrointestinal and lymphoid tissues. Crucially, all 8 organ variants show meaningful, relatively balanced activity — the RUI is used across a diverse range of tissue types, not dominated by a single organ focus. Male and female variants are used at comparable rates, confirming researchers use the anatomically correct models for their samples.

Exploration User Interface (EUI) — Trends & Feature Adoption

What is the EUI?

The EUI enables researchers to explore anatomical structures, cell types, and biomarkers through an interactive 3D organ viewer with hierarchical filter panels. Unlike the RUI (which is task-focused — register a tissue block), the EUI is discovery-oriented — researchers browse, filter, and query cross-dataset patterns. Key endpoint families: /api/v1/scene (3D viewer), /api/v1/cell-type-* (cell biology), /api/v1/ontology-* (anatomy), /api/v1/biomarker-* (molecular markers).

Fig 4 — EUI Monthly Usage: Event Volume & User Growth Trend
🔍 Key Insight — EUI User Growth Is the Story While raw event counts are relatively stable (10K–35K/month), the unique users chart is the critical signal: EUI unique users grew from ~450/month in early 2024 to ~2,900/month by December 2025 — a +200% year-over-year increase. This means the EUI is attracting more distinct researchers over time, even while individual session depth may be lower. This is a healthy growth signal: the EUI is becoming more widely adopted across the global HuBMAP research community.
Fig 5 — EUI Feature Adoption: Which Tools Are Actually Used?
🔍 Key Insight — Cell Biology Features Lead EUI Engagement Cell Type Filter (40K events) and Anatomical Structure Filter (29K events) are the most-used EUI features — researchers primarily use the EUI to query cell distributions and structure annotations, which is the platform's core scientific value proposition. All tree-navigation panels (Cell Type Tree, Ontology Tree, Biomarker Tree) are actively used (~21–26K events each), confirming hierarchical browsing is a central part of the EUI workflow. Provider/Lab Filter and Technology Filter (~16–27K events each) show researchers care about data provenance. Main 3D Scene calls (29K) confirm the 3D viewer is engaged but is a supporting tool, not the primary interaction layer.

RUI vs EUI — Comparative Platform Analysis

Fig 6 — Full Platform Comparison: RUI vs EUI vs CDE (Oct 2023–Jan 2026)
🔍 Key Insight — Two Very Different Usage Profiles The combined comparison reveals the core platform narrative:
  • RUI = Depth: High event intensity per session (tissue registration fires many API calls), bursty usage driven by events/releases, plateauing unique users (~1,000–1,400/month late 2025).
  • EUI = Breadth: Lighter event intensity per session (browse and filter), steady events but rapidly growing unique users. By Dec 2025, EUI surpasses RUI in unique monthly users.
  • CDE = Critical Gap: Consistently below 600 events/month across the full 28-month period. This represents a significant underutilization of advanced analytical tools.

Methods & Data Notes

Data Pipeline

All analysis is based on four CSV tables exported from Nachiket's DuckDB pipeline which processes the full 6.8M-event CloudFront Parquet log directly in-place (no full-table pandas load). The four tables are: monthly_usage.csv, feature_frequency.csv, traffic_summary.csv, geography_summary.csv.

Endpoint-to-Feature Mapping

Because CloudFront logs record HTTP requests (not UI click events), feature usage is proxied through API endpoint call frequency. For example, /api/v1/collisions is called every time a tissue block is moved in the RUI 3D viewer, making it a proxy for active placement interaction. The mapping is heuristic — a large slice of traffic still falls into "OTHER" (unclassified paths). Results should be interpreted as usage proxies, not exact click counts.

Interface Classification

RUI vs EUI vs CDE labels are assigned based on URL path patterns (e.g., paths containing collision or tissue-block → RUI; paths containing scene or cell-type → EUI; paths containing cde or histogram → CDE). Note: the absence of opacity-change endpoints in the logs indicates that opacity slider interactions are handled client-side in the browser and do not fire separate CloudFront requests.

Tools Used

Python · Pandas · NumPy · Matplotlib · Seaborn · DuckDB (via Nachiket's pipeline) · Jupyter Notebook
